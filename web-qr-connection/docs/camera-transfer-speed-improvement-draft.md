# カメラ転送の速度改善草案

作成日: 2026-05-20

## 目的

ネットワーク通信を使わず、画面表示とカメラ入力だけで画像やファイルを端末間転送するシステムにおいて、現行 QR chunk 方式の転送速度が極端に遅い問題を改善するための草案である。

ここでは実現可能性を厳密には絞らず、速度改善につながる可能性がある方式を広めに整理する。特に、QR を 1 枚ずつ読む方式から脱却し、1 回のカメラ入力で取得できる情報量を増やすことを重視する。

## 現状の速度ボトルネック

現行の QR 転送では、次の要因で転送速度が制限される。

- QR 1 枚あたりの実効 payload が小さい。
- base64 や JSON メタデータにより、実データ以外の文字数が増える。
- chunk を順番に集める必要があり、1 つ読み逃すと復元が止まりやすい。
- QR デコーダの検出速度、カメラ FPS、画面更新間隔に依存する。
- スマホを手で構えるため、角度、距離、手ブレで読み取りが不安定になる。
- QR の規格互換性を保つため、ファイル転送専用の高密度化がしにくい。

根本的な改善方針は、単に QR の表示速度を上げることではなく、次の 2 点である。

1. 1 フレームあたりに載せるデータ量を増やす。
2. 読み逃しがあっても復元できる転送方式にする。

## 改善方針

### 1. 画面全体を通信路として使う

QR のような小さなコードを 1 枚表示するのではなく、画面全体をデータ表示面として使う。

例:

- 画面全体を白黒またはカラーのセルに分割する。
- 各セルに 1 bit から数 bit を載せる。
- 四隅マーカーで位置と台形歪みを補正する。
- 受信側はカメラ画像から全セルを一括で読み取る。

これは既存調査の「独自カラータイル方式」に近いが、速度改善の観点では最も重要な方向である。QR の汎用仕様を捨て、ファイル転送専用の符号化にできるため、1 フレームあたりの情報量を大きく増やせる可能性がある。

### 2. 順番依存をなくす

連番 chunk をすべて順番に読む方式は、カメラ転送と相性が悪い。読み逃し、重複、ブレを前提にした方式へ寄せる。

候補:

- Fountain Code
- Erasure Code
- Reed-Solomon
- RaptorQ 風の冗長 block

受信側は「特定の chunk をすべて集める」のではなく、「復元に必要な数の block が集まったら完了」とする。これにより、欠落 chunk を待つ時間を減らせる。

### 3. 動画ストリームとして送る

静止コードを 1 枚ずつ読むのではなく、画面全体をデータ動画として連続表示する。

各フレームには次を含める。

- `sessionId`
- `frameId`
- payload block
- checksum または CRC
- FEC parity
- 同期パターン

スマホ側はカメラフレームを取り込み、読めた frame だけを蓄積する。すべてのフレームを完全に読む前提にせず、欠落と重複を許容する。

| 全画面データキャンバス | 高 | 高 | QR を捨て、画面全体を高密度データ面にする。本命候補。 |
| 動画ストリーム型データキャンバス | 高 | 高 | 全画面データキャンバスを連続フレーム化する。FEC と組み合わせると強い。 |

## 優先候補

### 第 1 候補: 複数 QR 同時表示

短期的に最も検証しやすい案である。

現在の QR chunk 構造を大きく変えず、1 画面に複数の QR を並べる。

例:

- 2x2 の 4 QR
- 3x2 の 6 QR
- 3x3 の 9 QR

スマホ側で複数 QR を同時検出できる場合、単純にスループットを上げられる。ただし、1 つあたりの QR が小さくなり、読み取り距離やピントの制約が厳しくなる。

検証すべき点:

- `html5-qrcode` または `zxing-wasm` で複数 QR を同時検出できるか。
- QR サイズを小さくしたときの認識率。
- 端末ごとのカメラ解像度とフォーカス性能の差。
- 1 QR 表示方式と比べた実効転送時間。

### 第 2 候補: 全画面データキャンバス

中期的な本命候補である。

画面全体を独自のデータ面として使い、QR より大きな情報量を 1 フレームに載せる。

初期 PoC では、次のような保守的な設定から始める。

| 項目 | 初期案 |
| --- | --- |
| セル | 白黒または 4 色 |
| グリッド | 32x18、40x24、64x36 などを比較 |
| 補正 | 四隅マーカー + 台形補正 |
| 同期 | frameId と固定 sync pattern |
| 誤り検出 | CRC |
| 復元 | 最初は FEC なし、次に erasure code を追加 |
| 評価 | セル認識率、フレーム認識率、実効 bytes/sec |

最初から高速化を狙いすぎると失敗原因が切り分けにくい。まずは静止フレーム 1 枚を正しく読めることを確認し、次に連続フレーム化する。

### 第 3 候補: Fountain Code / Erasure Code

QR 方式にも、全画面データキャンバス方式にも適用できる基盤改善である。

現在のように `index` と `total` を持つ連番 chunk だけでは、1 つの欠落が復元全体を止める。Fountain Code や Erasure Code を使うと、受信側は十分な数の block が集まった時点で復元できる。

期待効果:

- 読み逃した chunk の再表示待ちが減る。
- 表示順序に依存しにくくなる。
- カメラの一時的なブレやピント外れに強くなる。
- 動画ストリーム型の転送と相性が良い。

### 第 4 候補: 静止画ページ方式

リアルタイムスキャンをやめ、ファイルを複数枚の高密度データ画像に変換する方式である。

ユーザー体験:

1. PC 側にデータページを表示する。
2. スマホ側で撮影する。
3. 次のページへ進む。
4. 必要枚数を撮影したら、スマホ内でまとめて復号する。

リアルタイムデコードよりも、撮影後の高解像度画像解析に時間を使えるため、1 枚あたりのデータ密度を上げやすい。画像や大きめのファイル転送に向いている可能性がある。

### 第 5 候補: ローリングシャッター通信

最も実験的だが、高速化の可能性がある方式である。

スマホカメラは画像を一瞬で取得するのではなく、上から下へ順番に読み取るローリングシャッター方式であることが多い。PC 画面を高速に明滅させると、スマホのカメラ画像には時間変化が縞模様として記録される。

この性質を利用すると、画面の時間方向の信号を、カメラ画像上の空間パターンとして読める可能性がある。

検証すべき点:

- ブラウザ Canvas で十分な明滅周波数を出せるか。
- 一般的なスマホカメラで縞模様として観測できるか。
- 露出、シャッタースピード、リフレッシュレートの影響。
- 肉眼のチラつきが許容範囲か。
- データ復号に使える安定したパターンが得られるか。

## PoC の進め方

### Phase 1: 既存 QR の並列化

- 1 画面に複数 QR を表示する。
- 複数 QR を同時検出できるか確認する。
- 現行 1 QR 方式との転送時間を比較する。
- chunk サイズ、QR サイズ、表示間隔の組み合わせを測定する。

### Phase 2: 欠落耐性の導入

- 連番 chunk 前提をやめる。
- 冗長 block を生成する。
- 必要数が集まったら復元できる構造にする。
- 読み逃し率が高い状況での体感速度を比較する。

### Phase 3: 静止データキャンバス

- Canvas に白黒または 4 色のグリッドを表示する。
- スマホカメラ画像から四隅マーカーを検出する。
- 台形補正後にセルをサンプリングする。
- 1 フレームのセル認識率を測定する。

### Phase 4: 動画データキャンバス

- データキャンバスを連続フレームとして表示する。
- frameId、CRC、sync pattern を追加する。
- 読めたフレームだけを蓄積する。
- FEC と組み合わせて復元する。

### Phase 5: ローリングシャッター実験

- 白黒明滅または縞パターンの高速表示を試す。
- スマホカメラ画像に安定した縞が出るか確認する。
- 縞の幅、位相、明滅周期から bit を読めるか調べる。
- 実用候補に残すか、研究枠に留めるか判断する。

## 評価指標

方式ごとの比較では、単純な理論容量ではなく実効値を見る。

- 実効転送速度: bytes/sec
- 完了時間: 例として 10KB、100KB、1MB の転送時間
- フレーム認識率
- セル認識率
- chunk または block の欠落率
- 復元成功率
- ユーザーがスマホを構えている必要がある時間
- 距離、角度、手ブレへの耐性
- 端末差への耐性

## まとめ

転送速度を根本的に改善するには、QR を 1 枚ずつ順番に読む構造から脱却する必要がある。

短期的には複数 QR 同時表示で既存方式を並列化する。中期的には、画面全体を使うデータキャンバス方式と、読み逃しに強い Fountain Code / Erasure Code を組み合わせる。実験枠として、ローリングシャッター通信や輝度・周波数パターン符号化を検証する。

最終的な方向性としては、QR リーダー互換の汎用コードではなく、カメラ転送専用の Screen-Camera Communication プロトコルを設計するのが最も速度改善の余地が大きい。
